# 可选功能：Skill 使用提示

## 它解决什么问题？

开启后，Codex 在最终回复末尾用一行文字说明本轮实际使用了哪些 Skill，例如：

```text
本轮实际使用的 Skill：brief-mode（简洁回答）、systematic-debugging（系统化排错）
```

这样便于学习和检查：某个 Skill 是否真的参与了任务。

## 默认不启用，但会主动询问

这不是安装包，也不会修改本仓库任何 Skill。并非所有人都需要每轮看到提示，因此未经确认不启用；但当新用户让 Codex 按本仓库推荐或安装配置时，Codex 应主动问一次是否需要该功能。用户同意后，才由用户自己的 Codex 对**选定的本地 Skill**进行修改。

## 重要边界

- 这不是 Codex 内部的触发日志；Skill 本身无法读取产品内部的“已触发列表”。
- 它只能要求 Codex：在**实际读取并遵循**某个 Skill 流程后，如实自报名称。
- 仅看见 Skill 名称或元数据，不算实际使用，不应提示。
- 不要改 Codex 自带的 `.system` Skill：产品更新可能覆盖改动。
- 不要默认改全部 Skill；先让用户选择范围并确认。

## 让 Codex 帮你开启

把下面这段话发给 Codex：

> 我想开启“Skill 使用提示”。请先阅读本仓库的 `OPTIONAL-FEATURES.md`，列出我本地可修改的非 `.system` Skill，并建议最适合开启提示的范围。不要直接修改；等我确认具体 Skill 后，再给那些 Skill 加一条短规则：只有实际使用该 Skill 时，最终回复末尾合并标注“英文名（简短中文用途）”。不要修改 Skill 的触发条件、原有流程或 `.system` Skill。

## 新手配置时，AI 应怎样问？

在完成安装包推荐后、修改本地 Skill 前，主动用下面这句话询问一次：

> 还有一个可选功能：开启后，每次回复末尾会显示本轮实际使用的 Skill 和中文用途，方便你学习和检查。要开启吗？

用户同意才继续；拒绝或没有回答就跳过，不要反复追问。

## 建议的开启方式

1. **先从常用的 3–5 个开始**，例如简洁回答、调试、计划和验证类 Skill。
2. 用几次后，如果提示确实有帮助，再增加范围。
3. 如果觉得提示干扰阅读，让 Codex 删除各 Skill 中的“Skill 使用提示”小节即可恢复。

## 给 AI 的执行规则

当用户明确要求开启此功能，或在新手配置时回答同意时：

1. 先列出准备修改的 Skill、不会修改的 Skill 和原因。
2. 等待用户确认具体范围。
3. 只在所选 Skill 的 `SKILL.md` 正文中增加极短提示规则；不要改 YAML frontmatter（触发条件）。
4. 最终显示格式为：`本轮实际使用的 Skill：英文名（简短中文用途）`；多项时合并为一行。
5. 完成后检查：每个所选 Skill 恰好有一条提示规则，且 YAML frontmatter 未被破坏。
