# Brief Mode 中文说明

## 它是做什么的？

`brief-mode` 是一个用于控制 Codex 表达方式的个人 Skill。

它的目标不是让 Codex 永远少说，而是让回答更符合当前任务：

> 简单问题直接说重点；复杂、危险或需要动手操作的问题，必须讲清楚。

## 什么是 Skill？

Skill 可以理解成一份给 Codex 阅读的“工作规则”。当用户的请求符合触发条件时，Codex 会读取 `SKILL.md`，并按照里面的规则回答或执行任务。

`brief-mode` 主要解决这些问题：

- 回答铺垫太长，结论藏在后面；
- 一次给出太多选项，不方便逐个判断；
- 简单问题被写成长篇说明；
- 用户已经说“短一点”，回答仍然很啰嗦。

## 什么时候会使用？

出现下面这类表达时，它应该被触发：

- “短一点”或“重点说”；
- “TLDR”或“brief mode”；
- “别太啰嗦”；
- “一个个来”；
- “你讲得太多了”；
- “重点不明显”。

也可以直接说：

> 使用 `$brief-mode` 回答这个问题。

## 默认回答规则

1. 先给结论或决定。
2. 普通问题最多保留三个重点。
3. 能用 3～6 行讲清楚，就停止展开。
4. 用户继续追问时，只展开被问到的部分。

## 哪些情况必须详细？

下面这些任务不能为了简短而省略必要信息：

- 修改代码、文件或配置；
- 调试错误并分析证据；
- 删除文件或执行不可逆操作；
- 安装插件、依赖或 Skill；
- 涉及账号、权限、隐私、安全或费用；
- 给出必须准确执行的步骤；
- 用户明确要求“仔细讲”“展开”“一步一步来”或“完整方案”。

在这些情况下，`brief-mode` 仍然会先说结论，但会保留代码、证据、风险和操作步骤。

## 一次只讲一个

当用户在挑选 Skill、插件、工具或方案时，默认先介绍最推荐的一个，包括：

- 它是做什么的；
- 怎么使用；
- 最重要的优点和限制。

如果用户要求一个个来，就等待用户决定后再介绍下一个。

## 它会避免什么？

- 重复用户刚说过的问题；
- 大段铺垫、客套话和无关说明；
- 一次甩出很多相近选项；
- 对工具执行过程进行没有必要的直播；
- 与当前风险无关的通用免责声明；
- 明明已经回答完，又换一种说法重复结论。

## 简单示例

不理想：

> 我又找到一个和你需求比较接近的 Skill，它有很多特点，比如……然后我综合考虑以后认为……

更好：

> 找到一个更适合的：`claude-code-tldr`。它是短回答模式；我建议借鉴它制作自己的 `brief-mode`，不直接安装原版。

涉及风险或代码时则应展开：

> 结论：先备份配置，再修改。下面给出准确路径、修改内容和验证方法。

## 核心原则

> 该短就短，该详细时必须详细。
