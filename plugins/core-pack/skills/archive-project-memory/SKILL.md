# 归档项目记忆与 Notion 知识

用户发出触发语后，立刻归档本次对话的已确认结论；不需要等用户再次解释，也不把它当成普通聊天总结。

归档采用双通道判断：

- 项目 `memory/`：记录当前项目状态、进度、决定、失败项和下一步。
- Notion `codex知识库`：记录跨项目可复用的方法、教程、规则、决策模板和个人工作流。

根据内容价值自行判断写入一个或两个通道。不要把所有内容机械复制到两边。

## 执行步骤

1. 判断用户意图：
   - “总结一下”“总结总结”“归档本次”：执行智能双通道归档。
   - “只总结，不归档”：只在聊天中总结，不写文件或 Notion。
   - “只写项目 memory”：只写项目 `memory/`。
   - “整理到 Notion / Codex 知识库”：只写 Notion。
2. 提炼本次对话中已经确认的：目标、证据与结果、决定、失败或排除项、下一步。不要把猜测、尚未验证的方案写成结论。
3. 项目记忆归档：
   - 确认当前项目根目录；只在该项目内写入，绝不写入参考路径、临时目录或其他项目。
   - 若当前没有明确项目路径，不要伪造项目记忆；说明无法写入项目 `memory/`，继续判断是否适合写 Notion。
   - 若项目没有 `memory/`，创建它；更新最贴近主题的 `memory/*.md`。必要时新建主题文件，避免把所有历史塞进一个文件。
   - 创建或更新根目录 `MEMORY.md`，作为简短索引，链接到相关记忆文件并标注更新时间。
4. Notion 知识归档：
   - 只归档跨项目可复用内容：教程、方法论、评估标准、工作流、常见问题、通用决策记录。
   - 不把纯项目状态、临时进度、敏感细节或只对当前代码库有意义的流水账写入 Notion。
   - 优先搜索 `codex知识库` 中已有相关页面；能更新就更新，避免重复创建相似页面。
   - 如果 Notion 插件、授权或目标页面不可用，说明原因，并在聊天中给出可复制的 Notion 版本。
5. 回答用户：写入了哪些项目文件、写入或更新了哪些 Notion 页面、记录了什么、哪些内容因不适合或未确认而没有归档。使用通俗中文。

## 边界

- 这是显式触发的归档，不在普通对话中自动写入。
- 先读已有 `MEMORY.md` 和相关主题文件，合并更新，不重复或覆盖仍然有效的信息。
- 研究类项目必须保留实验条件、对照组和数值来源；没有证据的判断标为“待验证”。
- 归档本身不修改训练代码、配置、数据、评测标准或正式实验结果。
- Notion 是个人知识库，不替代项目 `memory/`；项目 `memory/` 是项目上下文，不替代 Notion 教程库。
- 用户明确限定某个通道时，尊重限定；不要自作主张写入另一个通道。
